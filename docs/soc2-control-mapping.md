
# SOC 2 Control Mapping

This document maps the implemented technical controls of the Storytime application to the relevant Trust Service Criteria (TSC) for a SOC 2 audit. It reflects the as-deployed state of the system.

--- 

## Common Criteria 6: Logical and Physical Access Controls

*   **CC6.1 (Logical Access):**
    *   **Control:** Access to data is restricted to authorized users.
    *   **Implementation:** Multi-tenancy is enforced at two layers. First, at the API layer, all Cloud Functions validate that the caller's authenticated UID matches the `ownerUid` of the data they are requesting. Second, at the database layer, Firestore Security Rules provide a defense-in-depth enforcement of the same UID ownership check. Direct data access is impossible.

*   **CC6.2 (User Authentication):**
    *   **Control:** All access requires authentication.
    *   **Implementation:** The system uses Firebase Authentication, which handles all user sign-up, sign-in, and session management. All API endpoints require a valid, unexpired JWT, which is validated on the server for every request.

*   **CC6.3 (Access for Privileged Roles):**
    *   **Control:** Access for privileged roles is restricted.
    *   **Implementation:** The system has no privileged roles. This control is satisfied by design, as there are no admin accounts or elevated permissions to secure.

*   **CC6.6 (Change Management):**
    *   **Control:** Changes to the system are authorized, tested, and implemented securely.
    *   **Implementation:** All code resides in a Git version control system. Deployment is automated via the `firebase deploy` command, creating a repeatable process. A full, passing test suite (including adversarial tests) is required before deployment, as demonstrated in the execution of this protocol.

*   **CC6.8 (System Monitoring):**
    *   **Control:** The system is monitored for anomalies.
    *   **Implementation:** All underlying infrastructure (Cloud Functions, Firestore, Firebase Hosting) is monitored by Google Cloud's native tools (Cloud Logging, Cloud Monitoring). All API requests and database interactions are logged by default, providing an audit trail for security reviews.

## Availability Criteria

*   **A1.1 (Capacity & Performance Monitoring):**
    *   **Control:** System performance is monitored to ensure availability.
    *   **Implementation:** The entire stack is serverless and managed by Google. Cloud Functions, Firestore, and Firebase Hosting automatically scale with demand. Google Cloud Monitoring provides dashboards for performance, execution time, and error rates.

*   **A1.2 (Disaster Recovery):**
    *   **Control:** A disaster recovery plan is in place.
    *   **Implementation:** The infrastructure is managed by Google, which has its own robust DR plans. The application itself is stateless. The source of truth (Git repository) and the deployment scripts (`firebase.json`, `package.json`) allow for a complete, rapid redeployment of the entire application to any region.

--- 

## Confidentiality Criteria

*   **C1.1 (Confidential Information Protection):**
    *   **Control:** Confidential information is protected from unauthorized access.
    *   **Implementation:** This is the core of the system's security model.
        1.  **Data at Rest:** Firestore encrypts all data at rest by default.
        2.  **Data in Transit:** All communication between the client, API, and database is over HTTPS/TLS.
        3.  **Cross-Tenant Prevention:** As described in CC6.1, strict, multi-layered ownership checks prevent any user from accessing another user's data.
        4.  **Asset Protection:** Raw assets in Cloud Storage are completely inaccessible, with security rules set to `allow read, write: if false;`. Access is granted *only* through short-lived, single-use Signed URLs generated by the backend for an authenticated, authorized user.
