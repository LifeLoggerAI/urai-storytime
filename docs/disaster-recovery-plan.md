# URAI-Storytime: Disaster Recovery Plan v1.0

## Overview

This document outlines the procedures to recover from potential disasters, ensuring data integrity and service availability for our users. A "disaster" is any event that causes significant data loss, corruption, or a prolonged service outage.

### Scenario A: Firestore Data Corruption or Loss

-   **Threat:** An accidental administrative error, a malicious act, or a software bug causes widespread data corruption or deletion within Firestore.

-   **Mitigation:**
    -   **Automated Backups:** We will enable and configure daily automated backups of the entire Firestore database.
    -   **Backup Retention:** Backups will be retained on a 7-day rolling basis, with a monthly snapshot retained for 30 days.

-   **Recovery Steps:**
    1.  **Isolate:** Immediately restrict write access to the database to prevent further corruption.
    2.  **Restore:** Initiate a database restore from the most recent, known-good snapshot.
    3.  **Validate:** Verify the integrity of the restored data.
    4.  **Notify:** If the data loss is significant and impacts user accounts, notify the affected families transparently.

### Scenario B: Cloud Storage Compromise

-   **Threat:** A bucket is misconfigured to be public, or service account keys with storage access are leaked.

-   **Mitigation:**
    -   **Secure by Default:** Storage buckets are configured to be private, with `storage.rules` denying all direct client reads and writes.
    -   **Signed URLs Only:** All content is delivered exclusively via short-lived (24h) signed URLs generated by our backend.
    -   **Access Logging:** Cloud Storage access logs are enabled to provide an audit trail.

-   **Recovery Steps:**
    1.  **Revoke:** Immediately revoke all active signed URLs and compromised credentials.
    2.  **Rotate:** Rotate all service account keys and administrative passwords.
    3.  **Audit:** Analyze access logs to determine the scope of the compromise.
    4.  **Notify:** If a data breach is confirmed, notify affected users and relevant authorities as per our Incident Response Playbook.

### Scenario C: Billing Desynchronization (Stripe <> Firestore)

-   **Threat:** A failed webhook or a bug causes a mismatch between a user's subscription status in Stripe and their access rights in Firestore.

-   **Mitigation:**
    -   **Idempotent Webhooks:** Our Stripe webhook handler is designed to be idempotent to handle duplicate events safely.
    -   **Reconciliation Job:** A daily automated job will be implemented to compare Stripe subscription statuses with Firestore billing states and flag discrepancies.

-   **Recovery Steps:**
    1.  **Sync:** Manually or automatically re-synchronize the correct subscription state from Stripe (the source of truth) to Firestore.
    2.  **Correct:** Update access rights for any affected families.
    3.  **Notify:** Proactively notify any families whose access was incorrectly impacted.

### Scenario D: Total Service Outage

-   **Threat:** A major upstream provider outage (e.g., Google Cloud region) or a critical deployment failure makes the platform unavailable.

-   **Mitigation:**
    -   **Cloud Monitoring:** We use cloud monitoring and alerting to detect service outages in real-time.
    -   **Auto-Scaling:** Cloud Functions are configured to autoscale, providing resilience against traffic spikes.

-   **Recovery Steps:**
    1.  **Rollback:** If caused by a bad deployment, immediately execute the rollback procedure to the last known stable version.
    2.  **Redeploy:** If necessary, redeploy the entire service from a clean, verified commit.
    3.  **Validate:** After recovery, perform a full system health check to ensure data integrity and correct billing states.
