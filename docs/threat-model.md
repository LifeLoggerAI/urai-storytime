
# Threat Model (STRIDE)

This document provides a threat model for the Storytime application based on the STRIDE framework. It identifies potential threats and documents the implemented mitigations.

--- 

## 1. Spoofing

*   **Threat:** A malicious actor impersonates a legitimate user to gain unauthorized access to their data and actions.

*   **Attack Vector:**
    *   Compromising a user's account credentials.
    *   Attempting to submit API requests with a forged identity.

*   **Mitigation:**
    *   **Firebase Authentication:** The system offloads all user authentication to Firebase Authentication, which provides secure, industry-standard OAuth 2.0 and OpenID Connect flows. This prevents credential theft via weak password policies.
    *   **JWT Validation:** Every request to the Cloud Functions API is required to have a valid JSON Web Token (JWT), which is rigorously validated by the Firebase Admin SDK on the backend. This ensures the identity of the caller is authentic.

## 2. Tampering

*   **Threat:** A malicious actor modifies data they are not authorized to change, such as another user's stories or child profiles.

*   **Attack Vector:**
    *   Calling the `generateStory` or `getStoryPlaybackDetails` functions with a `familyId` belonging to another user.
    *   Attempting to directly modify a Firestore document via a client-side SDK.

*   **Mitigation:**
    *   **Backend Validation:** As verified in the integration tests, every API function performs a server-side check to ensure the authenticated user's UID matches the `ownerUid` of the target `familyId` before performing any action. This is the primary defense.
    *   **Firestore Security Rules:** As a defense-in-depth measure, all Firestore collections are protected by security rules that enforce the exact same `ownerUid` check at the database level. An attacker who somehow bypassed the API would still be blocked by the database.
    *   **Immutability:** The Firestore rules for V1 explicitly set `allow update, delete: if false;`, preventing any modification or deletion of core data records, even by the legitimate owner.

## 3. Repudiation

*   **Threat:** A user denies having performed an action, and the system has no proof to the contrary.

*   **Attack Vector:**
    *   A user claims they did not create a specific story or family account.

*   **Mitigation:**
    *   **Ownership Records:** All key records (`families`, `children`, `stories`) are programmatically stamped with the `ownerUid` of the authenticated user who created them. This provides a strong, non-forgeable link between a user and their actions, sufficient for the current threat model.
    *   **(Gap):** For a higher level of compliance, a full immutable audit log could be implemented. This is noted as out of scope for the current freeze.

## 4. Information Disclosure

*   **Threat:** A malicious actor reads sensitive information belonging to another user.

*   **Attack Vector:**
    *   Attempting to read a Firestore document for a `familyId` they do not own.
    *   Gaining access to the path of a generated audio file in Cloud Storage and attempting to download it.

*   **Mitigation:**
    *   **Firestore Read Rules:** The `isFamilyOwner()` function in the `firestore.rules` file ensures that `read` operations are only permitted if the user is the owner of the data.
    *   **Deny-All Storage Policy:** The `storage.rules` are set to `allow read, write: if false;`. It is impossible for any user to directly read files from Cloud Storage.
    *   **Signed URLs:** The *only* way to access a file is via a short-lived, secure signed URL generated by the backend for an authenticated and authorized user. This URL grants temporary access to a single, specific file.

## 5. Denial of Service (DoS)

*   **Threat:** An attacker attempts to make the service unavailable to legitimate users.

*   **Attack Vector:**
    *   Flooding the API with a high volume of requests.

*   **Mitigation:**
    *   **Serverless Infrastructure:** The entire backend runs on Google's managed, auto-scaling infrastructure (Cloud Functions, Firestore). This provides significant resilience to traditional resource exhaustion attacks.
    *   **Google Cloud Armor:** Firebase Hosting and Google Cloud Functions are protected by Google's infrastructure, which includes DDoS mitigation.
    *   **(Gap):** Application-level rate limiting has not been implemented. This is a potential hardening improvement for a future version but is not required for the initial freeze.

## 6. Elevation of Privilege

*   **Threat:** A standard user gains the abilities of a privileged or administrative user.

*   **Attack Vector:**
    *   Attempting to access a non-existent administrative API endpoint.

*   **Mitigation:**
    *   **No Admin Roles:** The application has a flat permission model. There are no administrator roles or special privileges to be elevated to. A user is either the owner of a `family` record or they are not. This simple, robust model prevents this entire class of threat.
